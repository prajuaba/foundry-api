# Master Architecture & Functional Specification: Foundry.API Dynamic Engine

## 1. Goal & Architectural Philosophy
Design and implement a highly performance-optimized, zero-boilerplate, declarative API compilation engine named `Foundry.API`. This framework must serve as the universal execution orchestrator of the software factory. By consuming domain entities generated by `Foundry.Schema` and an operational behavior manifest (JSON), it must dynamically provision production-ready, highly secure, and optimized Service and Controller layers at runtime or compile time. It natively supports REST and GraphQL endpoints without requiring developers to write boilerplate business logic or mapping code.

## 2. Technical Stack & Execution Targets
- **Runtime Environment:** ASP.NET Core 8+ / .NET 10 Web API hosting environments.
- **REST Pipeline:** Optimized Minimal APIs leveraging source generators for route map optimization.
- **GraphQL Engine:** Hot Chocolate GraphQL server with structural BSON/POCO schema stitching.
- **Data Access Link:** Direct native injection and integration with `Foundry.Mongo` repository patterns.
- **Design Paradigms:** CQRS (Command Query Responsibility Segregation) pipelines orchestrated via MediatR behaviors.

## 3. Declarative Service Execution & Framework Features

### A. Dynamic Endpoint Materialization
- The engine must read an API configuration behavior manifest at application initialization.
- **REST Route Builder:** Dynamically map standard semantic routes (e.g., `POST /api/v1/orders`, `GET /api/v1/orders/{id}`) directly to standard generic `Foundry.Mongo` operations.
- **GraphQL Engine Integration:** Automatically expose target collections as queryable schemas, allowing clients to pass custom selections, field projection sorting arrays, and complex filtering syntax natively down to the driver level.

### B. Enterprise Cross-Cutting Pipelines (MediatR Behaviors)
Every generated inbound request must route through a decoupled, high-performance middleware execution pipeline enforcing four strict compliance boundaries:
1. **Validation Boundary:** Automatically discovers and executes structural property checks utilizing rules parsed from the schema definitions (via automated integration with `FluentValidation`).
2. **Security & Authorization Boundary:** Intercepts tokens, matches operation claims against claims matrices detailed in the configuration, and extracts the current context data payload into the ambient identity provider (`ICurrentUserContext`).
3. **Caching Layer:** High-speed execution intercepts for heavy idempotent queries, routing responses through distributed in-memory providers (e.g., Redis or HybridCache) with automatic cache invalidation triggers upon mutations.
4. **Audit and Resilience Boundary:** Ties directly into the underlying repository logging architecture to ensure exact request/response mapping logs correlate perfectly with database audit trail updates.

## 4. Documentation, Metadata, and OpenAPI Automation

### A. Auto-Generated OpenApi / Swagger Specifications
- The pipeline must use native `Microsoft.AspNetCore.OpenApi` configurations or `Swashbuckle` extensions to produce complete, structured Swagger/OpenAPI v3 definitions automatically.
- Every dynamically registered route must dynamically inject its request schemas, expected responses, error codes (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Internal Error), and security requirement schemas.

### B. Living Document Compiler
- Provide a runtime execution route (e.g., `/docs/spec`) that outputs a comprehensive technical architecture document package.
- It must assemble entity data rules, index optimization footprints, operational endpoint definitions, and active caching rules into a single visual summary document package for internal software auditing.

## 5. Performance, Memory Optimization & Diagnostic Resilience
- **AOT Friendliness:** Avoid heavy reliance on slow runtime reflection patterns. Use source generation engines (`Source Generators`) to inspect models at build time and write compile-time safe routing endpoints.
- **Asynchronous Flow Preservation:** Enforce a strict, unbroken async/await execution chain from the incoming HTTP context thread directly to the underlying MongoDB driver operations.
- **Structured Diagnostic Tracking:** Automatically instrument all dynamically generated endpoints with OpenTelemetry counters, tracing activities, and metrics tags to support instant integration with Datadog, Grafana, or Azure Monitor.

## 6. Deliverables Expected
1. **The Orchestration Manifest Core:** The definition scheme and parser managing the declarative API metadata configuration structures.
2. **The Dynamic Endpoint Provisioner Engine:** C# implementation patterns establishing route registrations and GraphQL type mappings at startup.
3. **The Global Request Middleware Pipeline:** Reusable components handling security checks, FluentValidation operations, and Telemetry logging blocks.
4. **Sample Application Footprint:** A clear sample application initialization flow displaying zero controller classes, a mock input behavior file, and automated Swagger UI generation.
